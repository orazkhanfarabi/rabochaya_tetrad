01 — Квиз: принципы работы Firestore

📚 ТЕОРИЯ Cloud Firestore — облачная документно-ориентированная база данных от Google, разработанная для масштабируемых мобильных и веб-приложений.

Ключевые возможности Firestore:
  • Реальное время — слушатели (addSnapshotListener) получают обновления мгновенно при изменении данных
  • Офлайн-поддержка — SDK кэширует данные локально, приложение работает без сети
  • Масштабируемость — автоматически масштабируется под любую нагрузку
  • Безопасность — Firebase Security Rules контролируют доступ на уровне документа

Типы запросов Firestore:
• get() — одноразовое чтение документа или коллекции
• addSnapshotListener() — подписка на изменения в реальном времени
• whereEqualTo(), whereLessThan(), orderBy(), limit() — фильтрация и сортировка

02 — Сортировка: операции записи в Firestore

📚 ТЕОРИЯ Операции записи в Firestore и их поведение:

set() — полностью перезаписывает документ. Если документа нет — создаёт. Все поля, не указанные в вызове, удаляются.

set() + merge:true — как set(), но существующие поля, которые не указаны, сохраняются (слияние).

update() — обновляет только указанные поля. Документ должен существовать заранее, иначе ошибка.

add() — создаёт новый документ с автогенерированным ID в коллекции.

delete() — удаляет весь документ целиком.

FieldValue.delete() — удаляет конкретное поле внутри документа при вызове update().

Расставь операции от «создания» к «удалению» в логическом порядке их применения в жизненном цикле документа:

⠿?
update() — обновить отдельные поля существующего документа
⠿?
add() — создать документ с автогенерированным ID
⠿?
delete() — удалить документ полностью
⠿?
FieldValue.delete() — удалить конкретное поле документа
⠿?
set() — записать или полностью перезаписать документ

03 — Вставь код: чтение и запись в Firestore

📚 ТЕОРИЯ Основные операции с Firestore на Kotlin/Android:

• Получить экземпляр: FirebaseFirestore.getInstance()
• Путь к документу: db.collection("name").document("id")
• Записать данные: documentRef.set(data) — принимает Map или data-класс
• Прочитать однократно: documentRef.get() — возвращает Task<DocumentSnapshot>
• Слушать изменения: documentRef.addSnapshotListener { snapshot, e → ... }
• Получить поле из снимка: snapshot.getString("fieldName")
• Фильтрация: collection.whereEqualTo("field", value).get()
• Добавить документ (авто-ID): collection.add(data)
💡 Слова для вставки: collection, document, set, addSnapshotListener, whereEqualTo
// Получение экземпляра Firestore
val db = FirebaseFirestore.getInstance()

// Ссылка на документ
val docRef = db.("users") .("uid_001")

// Запись данных
val data = mapOf("name" to "Alice", "age" to 25)
docRef.(data)

// Подписка на изменения в реальном времени
docRef. { snapshot, e ->
  val name = snapshot?.getString("name")
}

// Запрос с фильтром
db.collection("users") .("age", 25).get()

04 — Сопоставь метод Firestore с его назначением

📚 ТЕОРИЯ Методы Firestore и их точное назначение:
  • set(data) — полная перезапись документа переданными данными
  • update(data) — обновление только указанных полей, остальные не трогаются
  • add(data) — создание документа с автоматически сгенерированным ID
  • get() — одноразовое чтение документа или результатов запроса
  • addSnapshotListener() — подписка: callback вызывается при каждом изменении
  • orderBy(field) — сортировка результатов запроса по указанному полю

05 — Мультивыбор: Firestore на практике

📚 ТЕОРИЯ Правила безопасности и лучшие практики Firestore:

✅ Security Rules — правила пишутся на языке CEL и применяются на сервере без кода бэкенда
✅ Транзакции (runTransaction) — атомарное чтение и запись нескольких документов
✅ Batch write (WriteBatch) — запись до 500 операций одним атомарным вызовом
✅ Составные индексы нужны для запросов с несколькими условиями фильтрации/сортировки
✅ Пагинация реализуется через startAfter(lastDoc) + limit(n)
✅ Firestore тарифицируется по количеству операций чтения, записи и удаления
❌ Firestore НЕ поддерживает полнотекстовый поиск — нужен Algolia или Typesense
❌ addSnapshotListener нужно отписывать (remove()) при уничтожении компонента, иначе утечка памяти
❌ update() завершится ошибкой, если документа не существует
